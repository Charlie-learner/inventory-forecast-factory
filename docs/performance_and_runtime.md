# 执行耗时、性能优化与资源消耗分析

## 为什么原先需要等待较久

完整工作流不只是预测，还会调用 LLM 生成多个独立实现，并为每份源码启动隔离子进程，
执行安全、接口、确定性、边界输入和数值等价验证。联网研究与报告解释也可能各产生一次
外部调用。因此 API 延迟会被串行累加。

## 当前优化

- Web 使用 `/api/run-async` 创建后台任务，通过 `/api/jobs/{job_id}` 展示真实阶段和百分比；
- 多个独立代码实现按执行配置并行生成，生成后并行验证；
- 报告 Agent 只接收紧凑证据，不再发送完整 Trace 和源码；
- 正确性门禁通过后，按平均预测延迟、Python 内存峰值、源码规模和哈希确定性选优；
- 每次运行生成 `performance_analysis.json`，并写入技术报告和能力版本证据。

## 三种执行模式

| 模式 | 实现候选 | 并行度 | 性能基准次数 | 适用场景 |
|---|---:|---:|---:|---|
| `fast` | 1 | 1 | 10 | 日常试用、快速确认数据和需求 |
| `balanced` | 2 | 2 | 20 | Web 默认，兼顾等待时间与候选比较 |
| `thorough` | 3 | 2 | 40 | 深入分析、完整过程记录和代码方案比较 |

配置集中在 `inventory_agent/execution.py`，资源告警阈值集中在
`inventory_agent/services/performance.py`，不会散落在页面或具体算法中。

## 测量口径

隔离运行器对固定的 364 点历史和 28 天预测执行可配置次数的微基准，记录：

- 平均单次预测延迟；
- CPU 时间；
- 每秒预测调用数；
- `tracemalloc` 可见的 Python 分配内存峰值；
- 源码字节数、行数、AST 循环数和调用数。

`tracemalloc` 不保证覆盖 NumPy 等原生库的全部内存，因此报告明确将它定位为候选间相对
比较证据，而不是生产容量结论。生产发布仍应使用真实并发量、商品数和数据规模压测。

## 使用

```powershell
uv run python -m inventory_agent run `
  --description "预测商品 1001 在仓库 1 未来14天库存" `
  --data examples/business_data/demand_history.csv `
  --execution-mode fast
```

Web 高级设置可直接切换执行模式；任务启动后可以浏览其他页面，进度会继续轮询更新。

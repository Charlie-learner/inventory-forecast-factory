# 笔试项目最终交付审计

本文按《人工智能（大模型）-笔试.pdf》的交付内容、任务流程、开发要求、
进阶要求和四类评分标准逐项核查。结论只依据仓库内可运行代码、自动测试和已提交
产物，不把未来规划写成已完成功能。

## 一、总体结论

当前项目已经形成可演示、可复现、可审计的库存预测算法能力工厂原型：

1. 用户通过自然语言和数据文件提出库存预测或补货任务；
2. Agent 抽取外部文档或真实代码仓库中的算法能力；
3. 知识图谱结合需求画像检索候选算法；
4. 多候选执行无时间泄漏滚动回测；
5. 系统生成独立可运行代码并进行安全、接口、稳定性和数值等价验证；
6. 失败后分析根因、复用经验并进行多轮修复；
7. 生成面向业务用户和技术评审的两类报告；
8. 将验证结果、失败案例、修复策略和能力版本写回知识图谱。

可选联网研究节点还可检索 Crossref DOI 元数据，生成独立知识抽取文件，将来源写入图谱，
并为候选规划、代码生成和修复提供有边界的证据上下文。

在笔试评分框架内，基础要求和八项进阶要求均有可运行证据。主要剩余边界集中在
题目加分项的更高工程等级，例如容器级沙箱、Beam Search/MCTS、跨行业迁移和
生产部署联动。这些边界不阻断库存预测原型的完整闭环。

## 三、核心流程核查

### 1. 能力抽取

- 支持结构化 JSON、Markdown/TXT 和 Python AST；
- 支持递归扫描真实本地代码仓库；
- 抽取函数、类、参数、入口、直接及传递依赖、内部调用、复杂度和来源哈希；
- 外部资料记录标题、URL、许可、访问时间、置信度和证据位置；
- 抽取结果可直接摄取知识图谱。

边界：不负责自动克隆远程 Git 仓库，尚未覆盖跨语言调用图和任意 PDF/Word 语义抽取。

### 2. 能力复刻与代码生成

- Mock 模式使用安全、自包含模板保证离线可复现；
- API 模式根据能力规格、业务任务和需求画像，按多种策略生成多份独立源码；
- 每份实现保存策略、提示哈希、规格哈希、源码哈希和验证结果；
- 通过验证的实现按运行时间和代码规模自动选优；
- 最终源码直接实现算法，不是已有注册表的简单包装器。

边界：新颖且没有参考语义的算法仍需人工审核，模板外能力没有确定性回退实现。

### 3. 能力验证与修复

- 候选算法使用滚动起点回测，避免未来信息泄漏；
- 同时支持 WAPE、RMSE、SMAPE、Bias 和缺货/积压非对称库存成本；
- 生成代码经过语法、导入安全、统一接口、受限运行、稳定性、边界输入和数值等价检查；
- 失败被分类并生成稳定指纹、根因说明和建议动作；
- 最多两轮修复，成功经验可供后续同类失败检索复用；
- `examples/repair_run/` 保存“首次失败—修复—重新验证成功”的完整证据。

边界：当前为去密钥、超时子进程和 AST 白名单，不等同于容器或操作系统级沙箱。

### 4. 能力沉淀

- 图谱包含 Algorithm、SourceArtifact、DemandProfile、Metric、ValidationRun、
  FailureCase、RepairStrategy、CapabilityVersion 和 VersionEvent；
- 验证结果关联具体规格哈希和源码哈希；
- CLI/Web 支持版本查看、比较、发布和回滚事件；
- 历史验证成功率和库存成本会影响后续能力检索排序。

## 四、开发要求和交互核查

### Web、CLI 和 API

- Web 采用自然语言对话作为主入口，高级参数默认折叠；
- 支持一次上传多个 CSV，或上传 ZIP 数据包；
- 自动识别菜鸟全国表、分仓表和成本表，也支持普通字段别名与人工字段映射；
- 上传后显示可用商品 ID、仓库 ID 和数据质量提示；
- 支持多个商品、单仓、所有仓库和全国汇总语义；
- 运行完成后直接渲染业务报告，可切换技术报告、JSON、Trace 和产物清单；
- CLI 覆盖诊断、运行、抽取、复刻、回测、图谱、版本和插件；
- JSON API 提供自动生成的 OpenAPI 文档：`/api/docs` 和 `/api/openapi.json`。

### 面向业务用户的结果

当目录中存在 `inventory_snapshot.csv` 和 `replenishment_policy.csv` 时，系统不再把
预测量直接当作采购量，而是按以下业务信息计算建议补货：

- 当前现货、在途库存和欠单；
- 安全库存；
- 最小订货量和包装倍数；
- 仓库容量；
- 供应提前期；
- 商品/仓库级缺货与积压成本。

业务报告优先使用中文解释“一句话结论、建议怎么做、为什么、可信度和风险”；
WAPE、inventory_cost、代码门禁等专业内容保留在技术报告中。

## 五、评分标准对照

| 评分维度 | 高质量证据 | 当前风险 |
|---|---|---|
| 技术能力 40% | 完整 Agent 闭环、真实代码抽取、多代码实现、滚动回测、自动修复、图谱沉淀 | 无 MCTS；沙箱不是容器级 |

| 创新性 15% | 图谱增强检索、失败经验复用、能力版本血缘、成本感知库存决策、多实现代码选优 | 未做向量根因聚类和跨行业迁移 |
| 完整性 15% | Web/CLI/API、双报告、详细 Trace、知识图谱可视化、业务材料、提交示例和一键验收 | 公网鉴权、多租户和生产部署不在原型范围 |

`docs/advanced_requirements_audit.md`，评分证据见 `docs/scoring_alignment.md`。

uv run python scripts/validate_project.py

## 七、建议的现场演示顺序

1. 启动 Web，先展示自然语言主入口和完整业务示例；
2. 运行商品 1002、仓库 1 的 14 天预测，展示周季节画像与约束补货量；
3. 切换技术报告，展示候选算法回测、代码实现候选和验证门禁；
4. 查看运行 Trace，说明不同 Agent、工具调用和产物；
5. 打开 `examples/repair_run/`，展示自动失败分析与多轮修复；
6. 打开完整知识图谱，查看来源、算法、验证、失败、修复和版本血缘；
7. 展示本地代码仓库抽取和独立能力复刻；
8. 最后执行 `scripts/validate_project.py`，用自动化结果结束演示。

## 八、仍需如实说明的边界

- 项目是算法能力工厂原型，不会自动执行真实采购、调拨或生产发布；
- API/Web 面向本地评审演示，没有公网鉴权和多租户隔离；
- LLM API 模式受外部服务质量影响，Mock 模式用于确定性验收；
- 自动预测和补货结果必须结合活动、价格、供应异常和业务规则人工复核；
- 原始大数据集、个人密钥和运行时产物不应提交 Git。

这些边界不会削弱题目要求的核心闭环，反而使交付说明更可信、可审计。
